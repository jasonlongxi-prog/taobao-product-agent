# 淘宝采集与合规视觉制作：真实验收清单

## 1. 浏览器采集结果定位

Chrome下载目录可能被用户自定义，不要写死为Windows的 `Downloads`：

1. 先检查任务给出的目录。
2. 未找到时，读取Chrome当前配置中的下载位置或让用户查看 `chrome://settings/downloads`。
3. 在实际下载根目录下寻找 `TaobaoAgent/{item_id}/manifest.json`。
4. 找不到文件不等于采集失败；定位真实下载目录后再判断，避免要求用户盲目重复采集。

## 2. 清单与文件双重校验

不能仅凭弹窗“采集完成”宣称成功：

1. 读取 `manifest.json` 的 `counts`、`downloadSummary`、`images`。
2. 递归统计本地文件，并用Pillow验证每张图片可打开。
3. 对照清单数量、实际文件数、分类数和下载失败数。
4. 记录图片尺寸、格式、字节数和SHA-256；相同哈希只OCR一次。
5. GIF按首帧OCR，并在图片扩展名集合中显式纳入 `.gif`。

## 3. 动态标题不可信时

淘宝/天猫页面的宽泛选择器（尤其 `[class*=title]`）容易抓到“用户评价”“商品详情”等模块标题：

- 优先 `og:title`、`twitter:title`、`h1`、明确的商品标题组件和 `document.title`。
- 对“用户评价、评价、商品详情、宝贝详情、店铺、客服、推荐”等候选做拒绝过滤。
- 仍不可靠时写“标题待人工/OCR确认”，不要把短链接关键词冒充正式标题。

## 4. 图片分类与噪声清理

DOM启发式分类只能生成候选，不代表素材已经干净：

- 已知尺寸的图片若宽或高小于约180px，通常是按钮、图标、角标或占位图，应默认排除；SKU原图若自然尺寸较大则保留。
- 排除头像、Logo、sprite、loading、二维码、平台规则、价格说明、举报入口、无障碍、绿色治理等平台素材。
- 页面内嵌URL只有在邻近文本明显属于 `sku/prop/gallery/item/detail/desc` 数据时才纳入。
- 生成主图、SKU、详情三张联系表，用视觉检查真实商品图、页面UI、推荐商品和平台页脚。
- 平台营业执照不能当作卖家营业执照；必须区分平台主体、店铺主体和品牌授权。

## 5. OCR后的证据分级

分析报告必须区分：

- **图片直接事实**：设备外观、页面明确展示的功能文字。
- **页面营销宣称**：费率、流量、信号增强、续航、防水防摔、官方正品等。
- **合理推断**：可能适用的经营场景或用户痛点。
- **待核实**：正式标题、价格、具体SKU、授权、开通条件、支付方式、检测报告和售后规则。

重要结论标注原图文件名。OCR乱码不得补写成看似合理的参数。

## 6. 参考图重制为合规商品海报

目标是借鉴版式并保留真实产品，而不是一比一复制：

1. 识别参考图中的产品、标题、功能宣称、第三方Logo和版式。
2. 根据现有证据改写文案：产品类别准确；不承诺固定费率、永久免流量、百分百开通或绝对防漏单。
3. 删除未获授权的支付宝、微信、银联等第三方Logo；若支付能力已核实，也优先用文字列出并附适用条件。
4. AI图像编辑可用时：要求保留产品形态、移除旧文字和Logo、留出干净文字区；中文文字尽量后期本地叠加，避免模型错字。
5. AI图像编辑不可用时：用Pillow在原图无产品区域重绘背景和信息面板，保留真实产品区域，再用支持中文的字体叠加准确文案。
6. 若字体不支持项目符号导致方框字符，改为绘制圆形/几何项目符号，而不是继续使用缺字字形。
7. 输出后必须视觉复核：中文是否正确、设备是否被遮挡、旧Logo是否残留、分区是否有接缝、免责声明是否可读。
8. 最终明确说明是“基于用户真实产品图的合规重制”，不要伪称为全新AI生成产品实拍。

## 7. 推荐海报信息结构

- 主标题：准确的产品类别，例如“商家移动收款终端”。
- 利益副标题：可验证的用户收益，例如“收款提醒更清楚，对账管理更方便”。
- 功能点：实体按键、到账提醒、交易查询等，仅保留已核实内容。
- 底部确认项：支付方式、开通条件、设备费用、手续费、售后服务。
- 风险提示：具体功能以实际设备与服务协议为准；不承诺固定费率或“永久免流量”。
